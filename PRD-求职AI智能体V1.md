# 求职AI智能体 - 产品需求文档（PRD）

> **版本**：V1.0  
> **日期**：2026-05-16  
> **产品定位**：面向港中深商科研究生的 AI 求职助手，基于校友真实经历知识库，帮助学生明确求职方向、定位自身差距

---

## 一、产品概述

### 1.1 一句话定位
**用校友的真实求职路径，为在校生做 AI 导航。**

### 1.2 目标用户
| 维度 | 描述 |
|------|------|
| 学校 | 香港中文大学（深圳） |
| 学历 | 硕士研究生（研一/研二/研三） |
| 专业 | 商科为主（金融/会计/商业分析/管理等） |
| 痛点 | 求职方向迷茫、不清楚自身定位、不知道同背景学长学姐去了哪 |

### 1.3 产品形态
- **前端**：Web 网页应用
- **交互**：对话式 AI 智能体（Chatbot）
- **后台**：管理员上传校友访谈文档 → 系统自动解析入库

### 1.4 核心差异化
- 不以「通用求职建议」为基础，而是以**本校校友的真实经历**为数据源
- 用户画像不是静态表单，而是**每次对话后自动更新**的动态画像
- 长期记忆，能做到「你上次提到过想做投行，最近有进展吗？」

## 二、分阶段实施计划

```
Phase 1 ────────→  Phase 2 ────────→  Phase 3
简历识别 + 画像    校友知识库搭建     匹配引擎上线
(基础闭环)        (数据就绪)         (核心价值)
```

### Phase 1：简历识别 + 用户画像系统（MVP）
**目标**：用户能上传简历 → AI 解析 → 构建画像 → 对话中动态更新画像

**功能清单**：
- 简历 PDF/Word 上传 → AI 自动解析（教育背景、实习经历、技能、证书、目标方向）
- 解析结果展示 → 用户确认/对话修正
- 生成用户画像 JSON → 持久化存储
- 每次对话后，AI 自动提取新信息更新画像
- 对话历史向量化存储 → 下次对话时 RAG 检索注入上下文

### Phase 2：校友知识库搭建
**目标**：将 Word 访谈文档批量解析为结构化 JSON，建立可检索的校友知识库

**功能清单**：
- 后台管理页面上传校友访谈 Word 文档（支持批量拖拽上传）
- 批量上传时显示进度条、逐篇解析、单篇失败不影响整体
- AI 自动解析文档 → 提取结构化字段（见 5.4）
- 字段向量化 → 存入向量数据库
- 后台支持增/删/改/查校友记录
- 支持持续新增（增量入库）

### Phase 3：匹配引擎上线
**目标**：基于用户画像，检索最匹配的校友路径，提供个性化建议

**功能清单**：
- 画像 → 知识库相似度匹配
- 返回「最相似的 N 条校友路径」
- 简历诊断：对比目标岗位的成功校友简历 → 给出差距分析
- 求职规划：根据校友时间线，倒推用户每阶段该做什么

## 三、系统架构与解耦规则

### 3.0 核心原则

> **铁律：功能模块间通过标准化接口通信，严禁跨模块直接访问对方数据库/内部状态。**

```
┌─────────────────────────────────────────────────────────┐
│                     前端 Web 应用                        │
│              （对话界面 / 简历上传 / 画像展示）             │
└──────────┬──────────┬──────────┬──────────┬─────────────┘
│          │          │          │
┌─────▼────┐┌────▼─────┐┌───▼────┐┌───▼──────┐
│ 对话服务  ││ 画像服务  ││知识库服 ││ 匹配引擎  │
│ (chat)   ││(profile) ││ 务(kb)  ││(matcher) │
└─────┬────┘└────┬─────┘└───┬────┘└───┬──────┘
│          │          │          │
┌─────▼──────────▼──────────▼──────────▼─────┐
│              统一 API Gateway                │
│         （路由 / 鉴权 / 限流 / 日志）         │
└──────────────────┬─────────────────────────┘
│
┌──────────────────▼─────────────────────────┐
│              数据层（物理隔离）                │
│  ┌──────────┐ ┌──────────┐ ┌────────────┐  │
│  │ 画像DB   │ │ 知识库DB  │ │ 对话记录DB  │  │
│  │PostgreSQL│ │PostgreSQL│ │  + 向量库   │  │
│  └──────────┘ └──────────┘ └────────────┘  │
└─────────────────────────────────────────────┘
```

### 3.1 模块边界与接口契约

| 模块 | 职责 | 自有数据 | 对外暴露 API | 禁止事项 |
|------|------|---------|-------------|---------|
| **对话服务 (chat)** | 管理对话会话、消息收发、上下文编排 | 对话记录表、会话状态 | `POST /chat/send` `GET /chat/history/{session_id}` | ❌ 不直接读写画像库 ❌ 不直接查向量库 |
| **画像服务 (profile)** | 用户画像 CRUD、动态更新、置信度管理 | 用户画像表 | `GET /profile/{user_id}` `PUT /profile/{user_id}` `POST /profile/update-from-chat` | ❌ 不直接访问对话记录 ❌ 不调用匹配引擎 |
| **知识库服务 (kb)** | 校友文档解析、结构化存储、向量化、检索 | 校友记录表 + 向量库 | `POST /kb/documents` `GET /kb/search` `DELETE /kb/{alumni_id}` | ❌ 不访问用户画像 ❌ 不直接返回给前端（由对话服务编排） |
| **匹配引擎 (matcher)** | 用户画像 vs 知识库相似度计算、差距分析 | 无自有数据（纯计算层） | `POST /match/profile-to-alumni` `POST /match/gap-analysis` | ❌ 不存储任何数据 ❌ 不直接调用画像/知识库写操作 |
| **简历解析 (parser)** | PDF/Word → 结构化字段提取 | 无自有数据 | `POST /parse/resume` | ❌ 不存储解析结果（由画像服务写入） |

### 3.2 跨模块调用规则

```
✅ 允许的调用链                     ❌ 禁止的调用链

前端 → API Gateway → 任意服务      画像服务 → 直接查知识库DB
对话服务 → 画像服务 (API)          对话服务 → 直接写画像表
对话服务 → 知识库服务 (API)        匹配引擎 → 直接读用户画像表
对话服务 → 匹配引擎 (API)          知识库服务 → 对话记录表
匹配引擎 → 画像服务 (只读API)      任意服务 → 绕过Gateway直连
匹配引擎 → 知识库服务 (只读API)
画像服务 → 简历解析 (API)
```

### 3.3 数据隔离规范

| 规则 | 说明 |
|------|------|
| **每模块独立 Schema/DB** | 画像、知识库、对话记录分属不同数据库或至少不同 Schema |
| **无外键跨库** | 关联仅通过 `user_id` / `alumni_id` 字符串，不做数据库级外键约束 |
| **接口即边界** | 模块 A 需要模块 B 的数据，必须调 B 的 API，不能直连 B 的数据库 |
| **事件驱动更新** | 对话结束后发起 `chat.completed` 事件 → 画像服务异步更新，而非对话服务同步写画像 |

| 对话记忆（对话层） | 每轮对话向量化存入，下次对话 RAG 检索 | 第二周回来提到 "上次说的量化方向"，AI 正确引用历史 |

### 4.4 Sprint 3：知识库闭环（Week 5-6）⭐ Phase 2 上线

**目标**：后台管理页面上传 → Word 解析 → 结构化 → 向量化 → 可检索

| 知识库检索 | `GET /kb/search?q=投行+GPA3.5` | 返回 Top-N 匹配校友 |
| 后台管理页 | 上传文档 / 列表 / 编辑 / 删除 / 搜索 | 管理员可完整操作 |
| 对话服务接入 | 用户提问 → 对话服务调知识库检索 → 生成回答 | 「进投行需要什么」→ AI 引用了 3 条校友 insight 回答 |

> **Sprint 3 结束标志 = Phase 2 上线**：校友知识库可检索、对话中能引用真实校友数据。

### 4.5 Sprint 4：匹配闭环（Week 7-8）⭐ Phase 3 上线

**目标**：用户画像 → 匹配校友路径 → 差距分析 → 规划建议

> **Sprint 4 结束标志 = 全功能上线**：上传简历 → 画像 → 匹配校友 → 差距分析 → 规划建议，完整闭环。

### 4.6 开发铁律（每个 Sprint 必遵守）

| # | 规则 | 原因 |
|---|------|------|
| 1 | **先定接口，再写代码** | 每个 Sprint 开始前，前后端先对齐 API 契约（OpenAPI/Swagger） |
| 2 | **一个 Sprint 只动一个模块的核心逻辑** | 避免改画像服务顺手改知识库 → 耦合 |
| 3 | **新增字段必须通过 API，不扩展现有接口语义** | `GET /profile` 是读画像的，不要加个 `?include_kb=true` 让它顺便查知识库 |
| 4 | **每个模块独立部署/独立测试** | 知识库挂了，对话 + 画像仍可正常工作 |
| 5 | **数据迁移走独立脚本，不嵌在业务代码里** | 解析规则变了 → 跑迁移脚本重新入库，不改业务逻辑 |
| 6 | **Sprint 结束前，模块级集成测试全部通过** | 单元测试 + 模块联调测试，不积压到下一 Sprint |

## 五、核心功能详解

### 5.1 简历识别与画像构建（Phase 1）

**流程**：
```
用户上传简历（PDF/Word）
→ AI 提取结构化信息
→ 前端展示解析结果
→ 用户确认 or AI 对话引导修正
→ 写入用户画像 JSON
```

**AI 对话修正示例**：
> AI：「我从你的简历中看到你在华泰证券有一段投行部实习，时间是 2025 年 6-9 月，对吗？」  
> 用户：「对的。」  
> AI：「已确认。另外你提到想进咨询行业，但简历上没有咨询相关实习。需要我把这个作为你的提升方向记录下来吗？」

### 5.2 用户画像 JSON 结构

### 5.2.1 多目标画像与澄清机制

> 商科学生常同时准备 2-3 个方向（如投行 + 咨询 + 央企），画像必须支持多目标。

**多目标管理规则**：

| 规则 | 说明 |
|------|------|
| 每个目标有独立 `id` + `label` | 用户可命名：「投行方向」「咨询备选」「保底央企」 |
| `priority` 分 primary / secondary | primary 是主攻方向，匹配时优先展示 |
| `status` 标记准备状态 | `actively_preparing` 正在准备 / `exploring` 了解中 / `paused` 暂停 |
| `confidence` 标记确定程度 | 用户明确说「一定要做投行」= 0.9+，「可能试试 PE」= 0.5-0.6 |

**AI 澄清机制（对话中触发）**：

当用户表达模糊意向时，AI 主动澄清并归档：

> 用户：「我可能想做咨询或者投行，还没想好」
>
> AI：「明白，两个方向都是热门选择。我帮你都记下来——
>       🅰️ 咨询方向（主攻）
>       🅱️ 投行方向（备选）
>       后面匹配校友路径时，我会两个方向都搜，优先展示咨询的结果。
>       随着你了解更多，随时可以调整优先级。这样可以吗？」

**匹配时的多目标处理**：
- 用户未指定方向 → 两个目标各搜 Top-3，分别展示
- 用户指定方向（「我想看投行的」）→ 只搜该 target
- 用户说「都看看」→ 两个方向并列展示，标注匹配度


### 5.3 长期记忆机制（双层）

| 层级 | 存储形式 | 用途 | 更新时机 |
|------|---------|------|---------|
| **画像层** | 结构化 JSON | 快速了解用户全貌 | 每次对话后增量更新 |
| **对话层** | 向量化 embedding | 深度回溯历史上下文 | 每轮对话追加存入 |

**工作方式**：
- 用户打开对话 → 先加载画像 JSON（快速定位）
- 同时 RAG 检索近期相关对话 → 注入当前上下文
- 对话结束时 → AI 判断是否有新信息需要更新画像 → 增量写入

**对话检索场景举例**：
> 用户第二周回来：「上次你推荐的那个量化方向，我查了一下觉得挺有意思的。」

### 5.5 匹配引擎逻辑（Phase 3）

**匹配维度**（加权）：
| 维度 | 权重 | 说明 |
|------|------|------|
| 专业匹配 | 25% | 同专业优先 |
| 实习行业 | 30% | 同类实习经历 |
| 目标方向 | 25% | 想去的行业/岗位 |
| GPA区间 | 10% | 相近GPA |
| 技能重叠 | 10% | 证书、技术栈 |

**输出示例**：
> 「根据你的画像，找到 3 位背景相似的校友：  
> 1. **李四**（金融硕 2023 届），GPA 3.5，中信行研实习 → 中金投行 → PE 投资经理。**相似度 87%**  
> 2. **王五**（会计硕 2022 届），GPA 3.4，四大审计实习 → 华泰投行。**相似度 72%**  
> 3. ...」

### 5.5.1 空匹配降级策略

> 当没有任何校友与用户画像相似时，系统不能只返回「未找到匹配」。

**降级路径**（逐级放宽匹配条件）：

```
Level 0：精确匹配（默认）
    ↓ 无结果
Level 1：放宽专业限制（跨专业匹配同类实习/目标）
    ↓ 无结果  
Level 2：放宽目标行业（匹配相同专业、不同行业的校友）
    ↓ 无结果
Level 3：只看技能/证书重叠（不限专业和行业）
    ↓ 无结果
Level 4：全景浏览模式（展示知识库中最近 5 条校友路径，不限匹配度）
```

**AI 反馈话术**（按降级级别）：

| 级别 | AI 回复 |
|------|--------|
| Level 0 命中 | 「根据你的画像，找到 3 位背景相似的校友：...」 |
| Level 1 降级 | 「目前知识库中没有完全匹配的校友。我放宽了专业限制，找到了 2 位**目标行业相同**的校友，供你参考：...」 |
| Level 2 降级 | 「你的背景比较独特，数据库中暂时没有同方向的校友。不过有 3 位**同专业**的学长学姐，他们的求职经验也许对你有启发：...」 |
| Level 3 降级 | 「你的方向比较小众。我找到了 2 位**技能背景相似**的校友（都有 CPA / 券商实习），虽然行业不同，但准备路径可以借鉴：...」 |
| Level 4 兜底 | 「这个方向目前知识库中信息较少。我为你展示最近收录的 5 条校友路径，你可以浏览看看有没有启发。同时我会记录你的方向，后台会优先补充相关校友数据。」 |

**降级日志**：每次降级记录原因（无同专业 / 无同行业 / 无同技能），反馈给后台管理员 → 提示「该方向校友数据不足，建议优先补充」。


### 5.6 简历诊断（Phase 3）

用户上传简历 + 选择目标方向 → AI 检索该方向成功校友的「毕业时状态」→ 对比差距：

> 「你想进中金投行，对比 3 位成功入职的校友：  
> ✅ 你有：金融专业、华泰投行实习  
> ❌ 你缺：CPA（3位校友中2位有）、量化项目经验、内推渠道  
> 📋 建议：下学期考 CPA 一门 + 在 Kaggle 做一个金融建模项目」

## 七、对话策略设计

### 7.0 无简历场景：纯对话构建画像

> 针对研一新生或无简历用户，AI 通过结构化提问逐步构建画像。

**引导流程**：

```
AI: 「你好！我是求职导航助手。你可以直接上传简历，也可以先聊聊。
     你现在研几？学什么专业？」

用户：「研一，金融硕士」

AI: 「收到。本科是哪所学校、什么专业呢？」
用户：「中山大学金融」

AI: 「有没有实习经历？或者想往什么方向发展？」
用户：「还没有实习，但想做投行或者PE」

AI: 「好的，我记下来了。那技能方面呢？考过什么证书吗？编程水平怎么样？」
...
```

**引导规则**：

| 轮次 | 收集字段 | 追问策略 |
|------|---------|---------|
| 第 1 轮 | 学历 + 专业 | 直接问，最基础信息 |
| 第 2 轮 | 本科背景 | 顺藤摸瓜 |
| 第 3 轮 | 实习经历 / 目标方向 | 二选一，优先问目标（没实习也正常） |
| 第 4 轮 | 技能 + 证书 | 无论有无都问，空的就标记缺失 |
| 第 5 轮 | 地点偏好 / 薪资预期 | 可选，不想答就跳过 |

**关键原则**：
- 每轮只问 1-2 个字段，不过度追问
- 用户说「不知道」「没想好」→ 标记为空，不纠缠
- 画像构建完成后展示摘要，请用户确认
- 引导完成后主动提示：「以后有简历了随时上传，我会帮你更新画像」

### 7.1 首次对话（新用户引导）
```
AI: 「你好！我是求职导航助手。你可以直接上传简历，我会帮你分析。
或者先告诉我：你现在研几？想做什么方向？」

用户: [上传简历] 或 [文字描述]

AI: 解析简历 → 展示画像 → 逐项确认 → 问：「还有什么想补充的吗？」
```

### 7.2 后续对话（记忆驱动）
- 每次打开对话 → 加载画像 + RAG 检索相关历史
- 主动追问进展：「上次你说想考 CPA，报上名了吗？」
- 发现新信息 → 追问确认 → 更新画像

### 7.3 知识库问答
- 用户提问 → RAG 检索校友 insight → 回答附上来源（「根据 3 位校友的经验...」）

## 八、AI 角色人格定义

> **Persona 名称**：求职导航官（Career Navigator）
>
> **一句话**：像一位比你大两届、走过你这条路、愿意耐心陪你梳理方向的学长/学姐。

### 8.1 核心人格特质

| 特质 | 描述 | 示例 |
|------|------|------|
| **信息优先** | 先了解用户情况，再给建议，不凭空输出 | ❌「你应该去投行」→ ✅「你之前有过券商实习，投行方向可以继续深入，要不要看看类似背景校友的路径？」 |
| **引用驱动** | 有知识库数据时，必须引用校友真实经历，不编造 | 「根据 3 位金融硕校友的分享，中金投行面试最常问估值建模...」 |
| **承认边界** | 不知道就说不知道，不编造求职数据 | ❌「今年中金招了 15 人」→ ✅「具体招聘人数每年变化，建议关注官方校招公告」 |
| **引导不代劳** | 帮用户理清思路，不做人生重大决策 | ❌「你应该选投行」→ ✅「投行更看重 XX，PE 更看重 YY，你之前的经历更匹配投行，当然 PE 也可以补——你倾向哪边？」 |
| **适度追问** | 发现画像空白时主动问，但不连环追问 | 「刚才提到想考 CFA，报上名了吗？需要我帮你梳理备考时间线吗？」 |
| **温暖专业** | 语气温暖但不煽情，专业但不冰冷 | ❌「加油你一定行的！」✅「这条路有难度，但你目前的准备方向是对的，接下来重点是 XX。」 |

### 8.2 禁止行为

| 类别 | 禁止内容 | 原因 |
|------|---------|------|
| 🚫 编造数据 | 「今年投行平均薪资 50 万」「某公司招了 10 人」 | 除非知识库中有明确数据 |
| 🚫 替人决策 | 「你应该选这个 offer」「别去那家公司」 | 提供信息，决策权归用户 |
| 🚫 贩卖焦虑 | 「再不实习就来不及了」「你这个背景很难」 | 不制造焦虑，客观陈述差距 |
| 🚫 过度承诺 | 「按我说的做一定能进 MBB」 | 求职没有保证 |
| 🚫 评价用户 | 「你的 GPA 太低了」「你能力不行」 | 只陈述事实对比，不做价值判断 |
| 🚫 泄露隐私 | 向用户 A 透露用户 B 的信息 | 严格隔离 |

### 8.3 System Prompt 结构

```
[角色]
你是「求职导航官」，港中深就业中心的 AI 求职助手。
你的知识来自港中深商科校友的真实求职经历。

[用户画像]
{从画像 JSON 自动注入，包含学历/实习/技能/目标/差距}

[对话历史摘要]
{从对话向量库 RAG 检索注入}

[核心规则]
1. 所有求职建议必须基于校友知识库的真实数据，引用时标注「根据 X 位校友的经验」
2. 不知道的信息不要编造，直接说「这方面我没有足够的校友数据」
3. 引导用户理清思路，但不要替用户做决策
4. 发现用户画像中的空白字段，适度追问补充
5. 语气像一位走过这条路的前辈，温暖但专业

[本次对话]
{用户最新消息}
```

## 十二、对话质量评估体系

> **目的**：从 Phase 1 开始，持续监控 AI 对话质量，确保智能体不会「胡说八道」或「答非所问」。

### 11.1 评估维度

| 维度 | 权重 | 说明 | 评估方式 |
|------|------|------|---------|
| **准确性** | 30% | AI 回答内容事实正确（校友数据、求职常识不编造） | LLM-as-Judge + 人工抽检 |
| **相关性** | 25% | 回答是否切中用户问题，不跑题不啰嗦 | LLM-as-Judge |
| **记忆有效性** | 20% | 是否正确引用用户画像和历史对话 | 标注测试集验证 |
| **引导能力** | 15% | 是否能主动追问补齐画像空白，不被动等待 | 人工评分 |
| **语气得体** | 10% | 不生硬、不油滑、不越界（不替用户做重大决定） | 人工评分 |

### 11.2 评估流程

```
每次对话结束
│
├──→ 用户可选评分（⭐1-5）→ 存入对话记录
│
└──→ 定时任务（每周）
│
├── 随机抽取本周 10% 对话
│
├── LLM-as-Judge 自动评分（准确性 + 相关性 + 记忆有效性）
│
├── 低分对话（总分 < 3.0）→ 推送人工复审
│
└── 生成周报（各维度均分 + 趋势图 + 典型 bad case）
```

### 11.3 LLM-as-Judge 自动评分规则

> 使用独立 LLM 实例（非对话服务所用模型），对抽样对话逐条打分。

**Prompt 模板核心逻辑**：
```
你是求职AI助手的质检员。请对以下对话打分（1-5）：

1. 准确性：AI 的回答是否有事实错误？
- 特别关注：校友数据是否与知识库一致？通用求职建议是否正确？

2. 相关性：AI 是否直接回答了用户的问题？
- 扣分：用户问简历怎么改，AI 回答了一大段行业趋势

3. 记忆有效性：AI 是否正确引用了用户画像和历史？
- 扣分：用户上次说过 GPA 3.6，这次 AI 问「你 GPA 多少」

综合评分：____  问题标注：____
```

### 11.4 质量阈值与升级机制

| 状态 | 周均分 | 措施 |
|------|--------|------|
| 🟢 正常 | ≥ 3.8 | 维持现状 |
| 🟡 关注 | 3.5 - 3.7 | 查看 bad case，调整 prompt |
| 🟠 预警 | 3.0 - 3.4 | 暂停新功能开发，专项优化对话质量 |
| 🔴 严重 | < 3.0 | 系统下线检修，人工介入排查 |

### 11.5 人工复审抽样规则

| 触发条件 | 抽样比例 |
|----------|---------|
| 用户评分 ≤ 2 星的对话 | 100% 复审 |
| LLM-as-Judge 评分 < 3.0 的对话 | 100% 复审 |
| 每周常规抽检 | 随机 10 条（覆盖不同用户） |
| 新 prompt / 新模型上线后 48 小时内 | 50% 复审 |

### 11.6 质量看板（后台）

| 指标 | 展示形式 |
|------|---------|
| 周均分趋势 | 折线图（按维度分色） |
| 低分对话列表 | 表格，可点开查看完整对话 |
| 用户满意度分布 | 饼图（1-5 星占比） |
| Bad case 聚类 | 标签云（「编造数据」「答非所问」「遗忘上下文」等） |
| 模型成本 | 本周 API 调用次数 / Token 消耗 / 预估费用 |
